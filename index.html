ما المقصود بأداء نظام التشغيل؟

التعريف:

أداء نظام التشغيل يشير إلى كفاءة النظام في استخدام الموارد المتاحة (المعالج، الذاكرة، التخزين، الشبكة) لإنجاز المهام المطلوبة بأقل وقت واستهلاك للموارد.

أدوات المراقبة الشائعة:

1. مراقبة وحدة المعالجة المركزية:

الأدوات: top، htop، mpstat، sar

# مراقبة استخدام CPU لكل نواة
mpstat -P ALL 2
# التقرير: كل 2 ثانية عن جميع المعالجات

# عرض العمليات الأكثر استهلاكاً للـ CPU
top -o %CPU
# أو باستخدام htop (واجهة ملونة)
htop

2. مراقبة الذاكرة:

الأدوات: free، vmstat، /proc/meminfo

# عرض استخدام الذاكرة بشكل مقروء
free -h
# Output: total, used, free, shared, buff/cache, available

# إحصائيات الذاكرة والعمليات
vmstat 5
# تقرير كل 5 ثوانٍ

3. مراقبة القرص:

الأدوات: iostat، iotop، df، du

# مراقبة أداء أجهزة التخزين
iostat -dx 5
# تفاصيل موسعة كل 5 ثوانٍ

# مراقبة عمليات القرص في الوقت الحقيقي
iotop
# مشابه لـ top ولكن للقرص

4. مراقبة الشبكة:

الأدوات: iftop، nethogs، ss، netstat

# عرض حركة الشبكة في الوقت الحقيقي
iftop
# مشابه لـ top ولكن للشبكة

# إحصائيات واجهات الشبكة
ip -s link
# أو ss للاتصالات النشطة
ss -tunap

مثال متكامل للمراقبة

سيناريو: خادم ويب بطيء الاستجابة

نقوم بتشخيص المشكلة خطوة بخطوة باستخدام أدوات المراقبة

الخطوة 1: مراقبة الحمل العام

uptime
# الناتج:
10:30:00 up 20 days, 3:15, 3 users, load average: 4.5, 3.2, 2.8

# التفسير: load average مرتفع (4.5) على نظام ربما بـ 4 أنوية

الخطوة 2: تحديد العمليات الأكثر استهلاكاً للمعالج

ps aux --sort=-%cpu | head -10
# الناتج: قائمة بأكثر 10 عمليات استهلاكاً للـ CPU

top -bn1 | head -20
# أو استخدام top في وضع الدفعة

الخطوة 3: مراقبة استخدام الذاكرة

free -m
# الناتج:
total used free shared buff/cache available
Mem: 16048 12005 1024 512 3019 3542
Swap: 8192 4096 4096

# التفسير: الذاكرة ممتلئة بنسبة 75%، والـ swap مستخدم بنسبة 50%

الخطوة 4: مراقبة نشاط القرص

iostat -x 2 5
# الناتج (مبسط):
Device: rrqm/s wrqm/s r/s w/s rkB/s wkB/s avgrq-sz avgqu-sz await r_await w_await svctm %util
sda: 0.00 5.00 100.0 50.0 800.0 200.0 10.0 15.0 100.0 80.0 140.0 6.0 90.0

# التفسير: %util = 90% → القرص يعمل بكامل طاقته تقريباً

الخطوة 5: مراقبة الاتصالات الشبكية

ss -tunap | grep :80
# الناتج: عرض جميع الاتصالات على منفذ 80 (HTTP)

netstat -ant | grep :80 | wc -l
# عدد الاتصالات النشطة على منفذ 80

التشخيص النهائي:

  • المعالج: حمل مرتفع (load average: 4.5)
  • الذاكرة: ممتلئة بنسبة 75% مع استخدام swap
  • القرص: نشاط عالي (%util = 90%)
  • الشبكة: عدد كبير من الاتصالات النشطة

الخلاصة: الخادم يعاني من حمل زائد، يحتاج إلى تحسين الإعدادات أو زيادة الموارد

الجزء الثالث: تحسين أداء نظام التشغيل

استراتيجيات التحسين:

تحسين الأداء يجب أن يكون مبنياً على بيانات المراقبة وليس التخمين

1. تحسين استخدام المعالج

  • ضبط أولوية العمليات (nice values)
  • استخدام affinity لربط العمليات بمعالجات محددة
  • تفعيل تقنيات توفير الطاقة (Power Saving)
# ربط عملية بمعالجات محددة
taskset -c 0,1 /usr/bin/application

# تغيير governor للـ CPU
cpupower frequency-set -g performance

2. تحسين الذاكرة

  • ضبط swappiness (نسبة استخدام swap)
  • استخدام huge pages للتطبيقات الكبيرة
  • تحسين إعدادات الذاكرة الافتراضية
# تقليل استخدام swap
sysctl vm.swappiness=10

# تفعيل transparent huge pages
echo always > /sys/kernel/mm/transparent_hugepage/enabled

3. تحسين أداء القرص

  • اختيار نظام الملفات المناسب (XFS vs EXT4)
  • ضبط mount options (noatime, nodiratime)
  • استخدام RAID المناسب للحمل
  • تفعيل التخزين المؤقت (caching)
# في /etc/fstab
/dev/sda1 / ext4 defaults,noatime,nodiratime 0 1

# زيادة read-ahead
blockdev --setra 4096 /dev/sda

4. تحسين الشبكة

  • ضبط buffer sizes (TCP buffers)
  • استخدام TCP congestion control المناسب
  • تحسين إعدادات الجدار الناري
  • تفعيل TCP Fast Open
# تحسين إعدادات TCP
sysctl -w net.ipv4.tcp_window_scaling=1
sysctl -w net.core.rmem_max=16777216
sysctl -w net.core.wmem_max=16777216

قاعدة ذهبية في التحسين:

لا يمكنك تحسين ما لا تستطيع قياسه
- بيتر دراكر

قم بالقياس أولاً ← حدد المشكلة ← نفذ التحسين ← قم بالقياس مرة أخرى

دراسة حالة: تحسين خادم قاعدة بيانات

المشكلة:

بطء في استعلامات قاعدة البيانات MySQL، أوقات استجابة تصل إلى 5 ثوانٍ

الحلول المطبقة:

تحسين I/O:
# تغيير elevator algorithm لجدولة طلبات القرص
echo deadline > /sys/block/sda/queue/scheduler

# زيادة read-ahead value
blockdev --setra 8192 /dev/sda
تحسين الذاكرة:
# زيادة shared memory لنواة Linux
sysctl -w kernel.shmmax=2147483648
sysctl -w kernel.shmall=2097152

# تحسين إعدادات الذاكرة الافتراضية
sysctl -w vm.dirty_ratio=10
sysctl -w vm.dirty_background_ratio=5
ضبط limits:
# في ملف /etc/security/limits.conf
mysql soft nofile 65535
mysql hard nofile 65535
mysql soft nproc 16384
mysql hard nproc 16384

# في my.cnf (إعدادات MySQL)
innodb_buffer_pool_size = 8G
innodb_log_file_size = 1G
max_connections = 500

  • ضبط limits يمنع مشاكل نفاذ الموارد
  • التحسين المستمر بناءً على المراقبة

أدوات متقدمة للمراقبة والتحليل

1. Perf (Linux Performance Analysis)

أداة قوية لتحليل الأداء على مستوى النواة والتطبيقات

# تسجيل بيانات الأداء لبرنامج معين
perf record -g /path/to/program

# عرض التقرير
perf report

# تحليل استخدام system calls
perf trace -p [PID]

# قائمة جميع الأحداث القابلة للقياس
perf list

2. SystemTap

نظام تتبع وتصحيح ديناميكي لنواة Linux

# تتبع استدعاءات النظام لعملية معينة
stap -e 'probe syscall.open { printf("%s %s\n", execname(), argstr) }'

# مراقبة عمليات القرص
stap -e 'probe ioblock.request { printf("%s: %d\n", devname, size) }'

# قياس وقت تنفيذ دالة
stap -e 'probe kernel.function("vfs_read") { start = gettimeofday_us() }
probe kernel.function("vfs_read").return { printf("%d\n", gettimeofday_us() - start) }'

أسئلة للطلاب

السؤال الأول:

"إذا كان لديك خادم يحمل load average: 8.5, 7.2, 6.1 مع 4 أنوية معالجة، ماذا يعني هذا؟ وما هي الخطوات التي ستتخذها للتحقيق في المشكلة؟"

وقت التفكير: 120 ثانية

إجابة الطالب:

تحليل load average:

  • 8.5: متوسط الحمل خلال الدقيقة الأخيرة
  • 7.2: متوسط الحمل خلال 5 دقائق
  • 6.1: متوسط الحمل خلال 15 دقيقة

التفسير: النظام يعاني من حمل زائد بشكل كبير. الحمل 8.5 يعني متوسط 8.5 عملية في حالة runnable أو uninterruptible sleep، بينما لدينا فقط 4 أنوية.

خطوات التحقيق:

  1. تحديد العمليات الأكثر استهلاكاً:
    top -o %CPU أو ps aux --sort=-%cpu | head -10
  2. مراقبة استخدام الذاكرة:
    free -h و vmstat 2 10
  3. التحقق من نشاط القرص:
    iostat -x 2 5 أو iotop
  4. فحص عمليات الانتظار:
    pidstat -d 2 أو dstat
  5. مراقبة الاتصالات الشبكية:
    ss -tunap أو netstat -ant
  6. فحص logs للنظام:
    tail -f /var/log/syslog أو journalctl -f

الحلول المقترحة:

  • إنهاء العمليات غير الضرورية
  • ضبط أولويات العمليات (nice/renice)
  • إضافة موارد (ذاكرة إضافية، معالجات)
  • تحسين إعدادات التطبيقات
  • توزيع الحمل على خوادم متعددة

السؤال الثاني:

"ما الفرق بين nice و renice؟ ومتى تستخدم كل منهما؟ أعط أمثلة عملية."

إجابة الطالب:

الأمر nice:

الاستخدام: عند بدء تشغيل عملية جديدة

الوظيفة: تحديد أولويتها الابتدائية (nice value)

# تشغيل عملية بخلفية منخفضة الأولوية
nice -n 19 /usr/bin/backup.sh

# تشغيل عملية بأولوية عالية (تتطلب صلاحيات root)
sudo nice -n -20 /usr/bin/critical_process

المدى: من -20 (أعلى أولوية) إلى 19 (أدنى أولوية)

الأمر renice:

الاستخدام: لتغيير أولوية عملية قيد التشغيل

الوظيفة: تعديل أولوية عملية موجودة بالفعل

# تغيير أولوية عملية محددة (PID 1234)
renice -n 5 -p 1234

# تغيير أولوية جميع عمليات مستخدم معين
renice -n 10 -u username

المدى: نفس مدى nice (-20 إلى 19)

الفروقات الرئيسية:

المعيار nice renice
وقت الاستخدام عند البدء أثناء التنفيذ
التأثير على عملية جديدة على عملية موجودة
الاستخدام الشائع المهام الخلفية الطويلة التحكم في العمليات النشطة

سيناريوهات عملية:

  • استخدام nice: عند تشغيل backup ليلي - نعطيه أولوية منخفضة
  • استخدام renice: عندما نكتشف أن عملية تستهلك موارد كثيرة - نخفض أولويتها
  • استخدام كلاهما: تشغيل عملية بـ nice معين، ثم تعديلها لاحقاً بـ renice

السؤال الثالث:

"كيف تفسر هذه النتائج من أمر iostat؟ وما هي الإجراءات التصحيحية المقترحة؟"

Device: rrqm/s wrqm/s r/s w/s rkB/s wkB/s avgrq-sz avgqu-sz await r_await w_await svctm %util
sda: 0.00 5.00 100.00 50.00 800.00 200.00 10.00 15.00 100.00 80.00 140.00 6.00 90.00

إجابة الطالب:

تحليل المؤشرات:

  • %util = 90%: القرص يعمل بنسبة 90% من طاقته - مشكلة!
  • avgqu-sz = 15.00: متوسط طول طابور الانتظار 15 طلب - مرتفع جداً!
  • await = 100ms: متوسط وقت الانتظار 100 مللي ثانية - بطيء!
  • r/s = 100: 100 عملية قراءة في الثانية
  • w/s = 50: 50 عملية كتابة في الثانية
  • rkB/s = 800: 800 كيلوبايت قراءة في الثانية
  • wkB/s = 200: 200 كيلوبايت كتابة في الثانية

التشخيص:

القرص sda يعاني من:

  1. ضغط شديد (%util = 90%)
  2. طابور انتظار طويل (avgqu-sz = 15)
  3. أوقات استجابة عالية (await = 100ms)
  4. نشاط قراءة مرتفع (100 عملية/ثانية)

النتيجة: القرص أصبح bottleneck للنظام بأكمله!

  • زيادة حجم buffer للقراءة/الكتابة
  • تفعيل التخزين المؤقت في التطبيق
تحسين النظام:
  • ترقية إلى أقراص أسرع (SSD بدلاً من HDD)
  • استخدام RAID 10 لتحسين الأداء
  • زيادة ذاكرة التخزين المؤقت
  • ضبط elevator scheduler
تحسين نظام الملفات:
  • استخدام mount options: noatime,nodiratime
  • اختيار نظام ملفات سريع (XFS, ext4)
  • ضبط journaling settings
حلول أخرى:
  • تقسيم البيانات على أقراص متعددة
  • استخدام memory-mapped files
  • تفعيل compression للبيانات

أمثلة عملية:

# تغيير scheduler للقرص
echo deadline > /sys/block/sda/queue/scheduler

# زيادة read-ahead
blockdev --setra 8192 /dev/sda

# في /etc/fstab
/dev/sda1 /data ext4 defaults,noatime,nodiratime,data=writeback 0 2

الخاتمة والنقاط الرئيسية

النقاط الرئيسية:

  1. إدارة الأداء عملية مستمرة وليست مرة واحدة
  2. المراقبة المستمرة ضرورية للكشف المبكر عن المشاكل
  3. التحسين يجب أن يعتمد على بيانات واقعية وليس تخمين
  4. كل نظام له خصوصيته؛ لا يوجد حل واحد يناسب الجميع
  5. القياس قبل وبعد التحسين ضروري لتقييم الفعالية

الشعار الذهبي:

"لا يمكنك تحسين ما لا تستطيع قياسه"
